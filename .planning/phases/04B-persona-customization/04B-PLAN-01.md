## Objective

Apply the persona table migration to Supabase using the MCP tool `mcp__supabase__apply_migration`. This is a one-time DDL operation that must be confirmed by a human before execution because it directly modifies the remote project schema. The update_updated_at() trigger function is assumed to already exist (used by other tables such as memories).

## Tasks

### Task 1: Verify update_updated_at trigger function exists

<read_first>
- supabase/migrations/ (reason: check whether update_updated_at is defined in an existing migration to confirm it is safe to reference)
</read_first>

<action>
Before applying the migration, confirm that the function update_updated_at() already exists in the Supabase project by running:
SELECT routine_name FROM information_schema.routines WHERE routine_schema = 'public' AND routine_name = 'update_updated_at';

If the function does not exist, add a CREATE OR REPLACE FUNCTION block at the top of the migration before the persona table DDL.
</action>

<acceptance_criteria>
- Query returns exactly one row with routine_name = 'update_updated_at', OR the migration SQL includes the function definition
</acceptance_criteria>

### Task 2: Apply migration via mcp__supabase__apply_migration

<read_first>
- No file reads required — SQL is fully specified below
</read_first>

<action>
Call mcp__supabase__apply_migration with the following SQL verbatim:

CREATE TABLE public.persona (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id      UUID NOT NULL UNIQUE REFERENCES public.users(id) ON DELETE CASCADE,
  companion_name TEXT NOT NULL DEFAULT 'Arjun',
  tone         TEXT,
  expectation  TEXT,
  open_field   TEXT,
  created_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at   TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE TRIGGER persona_updated_at
  BEFORE UPDATE ON public.persona
  FOR EACH ROW EXECUTE FUNCTION update_updated_at();
CREATE INDEX idx_persona_user_id ON public.persona(user_id);
ALTER TABLE public.persona ENABLE ROW LEVEL SECURITY;
CREATE POLICY "persona_own" ON public.persona
  FOR ALL USING (auth.uid() = user_id);

Name the migration: 20240001_create_persona_table
</action>

<acceptance_criteria>
- mcp__supabase__apply_migration returns success with no error
- Running the verification SQL from must_haves.commands shows all 8 expected columns
- pg_policies shows policyname = 'persona_own' for tablename = 'persona'
- pg_indexes shows idx_persona_user_id for tablename = 'persona'
</acceptance_criteria>

### Task 3: Smoke-test insert and RLS

<action>
Using mcp__supabase__execute_sql, run a quick sanity check:

SELECT COUNT(*) FROM public.persona;

Expect result: 0 rows (empty table, not an error).

Also verify the constraint by attempting:
INSERT INTO public.persona (user_id, companion_name) VALUES (gen_random_uuid(), 'Test');
Then DELETE FROM public.persona WHERE companion_name = 'Test';

Confirm the insert and delete both execute without FK violation (the gen_random_uuid() user_id will violate the FK — this confirms the FK constraint is active). The insert is expected to FAIL with a foreign key violation error.
</action>

<acceptance_criteria>
- SELECT COUNT(*) returns 0
- INSERT attempt fails with "violates foreign key constraint" referencing public.users(id)
- This confirms the FK is correctly wired
</acceptance_criteria>
